# alipay-preliminary-design 技能指南

## 概述

本技能用于指导用户撰写符合技术规范的技术方案概要设计文档。文档遵循五大段落结构，保持朴素平实的行文风格，重点描述"是什么"和"为什么"，不过度展开技术细节。

## 技能使用说明

### 触发条件
当用户需要撰写以下类型的技术方案文档时，使用本技能：
- 系统架构方案
- 技术实现方案
- 技术改进方案
- 基础设施方案

### 文档范围
- **文档类型**：概要设计（非详细设计）
- **核心输出**：符合规范的技术方案文档
- **使用工具**：
  - 撰写: 本技能提供的指导和模板
  - 架构图: [draw-io skill](./references/diagram-tools.md)
  - C4图: [c4-architecture skill](./references/diagram-tools.md)

## 五大段落结构详解

### 1. 需求章节
**详略原则**：允许详细描述

**包含内容**：
- 业务背景和目标
- 功能需求清单
- 非功能需求（性能、安全、可靠性等）
- 约束条件和假设

**写作要点**：
- 清晰描述问题的全貌
- 明确项目的边界和范围
- 列出关键业务指标和技术指标

### 2. 分析章节
**目标**：从需求提炼关键挑战点和技术难点

**分析维度**：
- 技术挑战（性能、扩展性、兼容性等）
- 业务挑战（流程复杂性、合规要求等）
- 资源挑战（时间、人力、成本等）

**输出**：清晰的问题列表，为解决方案提供靶点

### 3. 解决方案章节
**原则**：针对分析章节提出的挑战点给出解决方案，保持概要化

**结构要求**：
- **方案筛选机制**：
  - 提供多种可选方案
  - 进行方案对比（优缺点分析）
  - 给出决策依据和推荐理由
- **图表配套要求**：
  - **每个推荐方案必须附带配套的架构图和C4图**
  - 使用 draw-io skill 绘制整体架构图
  - 使用 c4-architecture skill 生成C4模型图（系统上下文、容器、组件图）
  - 图表应清晰展示方案的技术架构和组件关系
- **详略控制**：
  - 描述"是什么"和"为什么"
  - 不过度展开"怎么做"的具体细节
  - 关键技术点可简要阐述

### 4. 待确认项章节
**内容**：
- 解决方案中还未彻底解决的挑战点
- 需要多方讨论确认的业务规则
- 悬而未决的架构决策

**重要原则**：关于技术选型等待确认项，遵循"先询问用户，后列入清单"的原则：
- 当遇到需要额外信息才能确定的技术选型时，应当优先主动询问用户
- 若用户能够当即提供有效补充信息，则直接在文档中明确技术选型
- 若用户表示无法当即提供有效信息，则将其列入"待确认项"章节
- 所有待确认项都需要提供可选方案或推进建议

**呈现方式**：
- 清晰列出每个待确认项
- 说明为什么需要确认
- 提供可选方案或建议

### 5. Tasks/实施计划章节
**制定原则**：基于已确定的解决方案

**包含要素**：
- 任务分解（Task breakdown）
- 优先级排序
- 时间估算
- 依赖关系
- 资源分配
- 关键里程碑

## 文档格式规范

### Markdown 格式要求
- 使用标准 Markdown 语法
- 合理使用标题层级（#、##、###）
- 使用列表、表格增强可读性
- 关键信息加粗或使用引用块

### 目录层级限制
- **原则**：最多2层结构（如 2.1、3.2）
- **例外**：需求章节可突破此限制
- **禁止**：除需求章节外，不得出现 2.1.1、3.2.1 等3层及以上层级

### 文件命名规范
```
主文档：项目名称-preliminary-design-YYYYMMDD.md
架构图：项目名称-architecture.drawio
C4图：项目名称-c4-diagram.md
资源文件：项目名称-资源类型-描述.扩展名
```

### 图片和资源引用
- 文件内直接引用图片或资源
- 若 Markdown 文本中不方便直接引用：
  - 添加占位符说明
  - 说明资源文件名称和位置
- 示例：`![架构图](./assets/architecture.drawio)` 或 `【架构图待补充：见 assets/architecture.drawio】`

## 图表工具使用指南

### draw.io skill 使用方法
**用途**：绘制整体架构图

**输出格式**：drawio 文件格式

**使用流程**：
1. 调用 draw-io skill
2. 描述架构图需求
3. 保存输出的 drawio 文件到 assets/ 目录
4. 在文档中添加引用或占位符

**最佳实践**：
- 分层绘制（基础设施层、服务层、应用层）
- 使用标准图标和符号
- 保持版本一致性

### c4-architecture skill 使用方法
**用途**：生成 C4 架构图（系统上下文、容器、组件图）

**输出格式**：Markdown 文件格式（包含 Mermaid 图表）

**使用流程**：
1. 调用 c4-architecture skill
2. 提供系统边界和组件信息
3. 保存输出的 Markdown 文件
4. 在主文档中引用或嵌入

### 图表版本管理
- 每次文档修改后，必须重新生成相关图表
- 确保文档内容与图表版本一致
- 主文档与配套图表统一存放

## 文档风格要求

### 总体风格定位
- **行文风格**：朴素、平铺直叙、接近技术报告或论文风格
- **语言特点**：准确、有条理、不浮夸
- **避免**：冗余描述和过度技术细节堆砌

### 详略控制原则
**重点描述**：
- "是什么"：方案的整体设计
- "为什么"：设计决策的理由和依据

**适度描述**：
- "怎么做"：关键技术点的实现思路

**避免描述**：
- 详细的代码实现
- 完整的配置文件
- 逐步的操作指南

### 动态详略原则
- **需求章节**：可详细描述，充分展开
- **分析章节**：提炼关键点，简明扼要
- **解决方案**：概要化描述，突出重点
- **待确认项**：清晰罗列，说明原因
- **实施计划**：明确具体，可执行

### 示例对比
❌ 过度详细（不推荐）：
```markdown
### 3.1 缓存方案
使用 Redis 集群作为缓存，配置如下：
- 节点数：3 主 3 从
- 配置参数：maxmemory 4GB, maxmemory-policy allkeys-lru
- 连接池：最大连接数 100，超时时间 2000ms
- 具体代码实现：
  ```java
  @Configuration
  public class RedisConfig {
      @Bean
      public RedisTemplate<String, Object> redisTemplate() {
          // 详细配置代码...
      }
  }
  ```
```

✅ 概要化（推荐）：
```markdown
### 3.1 缓存方案
采用 Redis 集群解决高并发读取性能问题。选择 Redis 的原因：
1. 支持高并发访问，符合 QPS 10万+ 的需求
2. 数据持久化机制完善，确保数据可靠性
3. 社区活跃，运维成本低

集群规模根据数据量和访问量评估，具体配置在实施阶段确定。
```

## 内容识别与标注规范

### 功能状态标注
所有功能点必须明确分类并标注：

#### 已实现功能
**标注方式**：
- 在文档中明确说明 "已实现"
- 弱化或删除相关"困难描述"
- 如需说明，侧重当前状态而非问题解决过程

**示例**：
```markdown
### 2.3 用户认证
✅ **已实现**：基于 OAuth 2.0 的统一认证系统已于 2024 Q1 上线
- 支持支付宝账号、手机号、生物识别等多种认证方式
- 日活用户 500 万+，99.9% 可用性
```

#### 待实现功能
**标注方式**：
- 使用将来时态描述
- 概要阐述功能和目标
- 简单说明关键技术方案，不列详细实现

**示例**：
```markdown
### 3.2 异步消息队列
**待实现**：引入消息队列实现服务解耦和流量削峰
- 功能目标：支持订单状态异步通知，提升系统响应速度
- 技术选型：RabbitMQ（评估中，待确认）
- 预期效果：接口响应时间降低 50%，提升用户体验
```

#### 待讨论项
**标注方式**：
- 明确标注为"待讨论确定项"
- 列明多种可选方案
- 说明各方案优缺点
- 必须在"待确认项"章节中汇总

**示例**：
```markdown
### 3.3 数据库选型
🔍 **待讨论确定项**（方案对比）

| 方案 | 数据库 | 优点 | 缺点 | 适用场景 |
|------|--------|------|------|----------|
| A | MySQL | 事务支持完善、生态成熟 | 扩展性受限 | 交易数据 |
| B | MongoDB | 灵活的数据模型、易扩展 | 事务支持弱 | 日志数据 |
| C | TiDB | HTAP、水平扩展 | 运维成本高 | 混合场景 |

**推荐**：倾向方案 A，待技术评审确认
```

### 待确认项章节格式
```markdown
## 4. 待确认项

### 4.1 技术选型待确认
1. **数据库选型**：见章节 3.3
   - 影响：数据模型设计、系统扩展性
   - 建议：组织技术评审会，邀请 DBA 团队参与

### 4.2 业务规则待确认
2. **订单状态同步机制**
   - 问题：实时同步 vs 定时同步的选择
   - 影响：系统复杂度、数据一致性
   - 建议：与产品团队讨论业务容忍度
```

## 技术规范要点

### 方案筛选机制
**决策流程**：
1. **需求匹配度**：方案是否满足核心需求
2. **技术可行性**：团队技术栈、实施难度
3. **成本评估**：开发成本、运维成本
4. **风险评估**：技术风险、业务风险
5. **扩展性**：未来业务增长支持

**文档呈现方式**：
- 提供方案对比表格
- 提供决策矩阵
- 给出明确的推荐理由

### 对比和决策依据
**对比维度**：
- 功能性：满足需求的程度
- 性能：响应时间、吞吐量、扩展性
- 可靠性：可用性、容错能力
- 成本：开发、运维、硬件
- 风险：技术成熟度、团队熟悉度
- 生态：社区支持、文档完善度

**决策依据呈现**：
- 数据支撑：性能测试数据、成本测算
- 经验依据：业界最佳实践、团队经验
- 原则遵循：架构设计原则、技术选型标准

## 内容质量标准

### 信息准确性要求
- **数据准确性**：所有数据、指标、配置需准确无误
- **来源清晰**：提供数据来源或计算依据
- **具体化**：避免模糊描述，使用具体数值和范围

**示例**：
- ❌ "系统性能大幅提升"
- ✅ "系统响应时间从 500ms 降低至 200ms"

### 逻辑完整性检查清单
- [ ] 需求与分析对应：每个分析项都能找到需求来源
- [ ] 分析与解决方案闭环：所有分析的挑战都有解决方案或待确认项
- [ ] 技术决策可追溯：重要决策都有依据和理由
- [ ] 待确认项可追踪：每个待确认项都有责任人和跟进计划
- [ ] 实施计划可执行：Tasks 分解合理，时间估算准确

### 可读性优化建议
- **结构清晰**：使用合理的标题层级
- **重点突出**：关键信息使用列表、表格
- **段落适中**：避免过长的段落（建议不超过 8 行）
- **句式简洁**：使用简单句，避免复杂嵌套
- **图表辅助**：复杂逻辑用图表表示

## 常见问题与最佳实践

### Q1: 需求章节应该写多详细？
**A**: 需求章节是五大段落中唯一可以详细描述的章节。建议：
- 充分描述业务背景和问题
- 详细列出功能点和非功能需求
- 明确项目边界和假设
- 提供相关的业务流程图

### Q2: 解决方案章节如何避免过于详细？
**A**: 把握"概要"的核心：
- 描述架构和核心设计思路
- 说明关键技术选型及原因
- 阐述设计决策的依据
- 不编写详细代码和配置
- 不提供分步实施指南

### Q3: 如何判断一个项是"待确认项"还是直接给出解决方案？
**A**: 判断标准：
- 需要更多信息才能决策 → 待确认项
- 需要多方讨论才能确定 → 待确认项
- 对系统有重大影响且存在多种选择 → 待确认项
- 方案明确且风险可控 → 直接给出解决方案

### Q4: 文档中如何平衡准确性和时效性？
**A**: 处理原则：
- 确定性内容：准确描述，如已上线功能、明确需求
- 变化中内容：标注状态和时间，如"截至 2024-02-01"
- 待确认内容：明确标注，如"待讨论确定项"
- 规划性内容：使用适当限定词，如"计划"、"预期"
