## 结构性检查

### 1.1 五大段落完整性

- [ ] **需求章节存在**：包含业务背景、功能需求、非功能需求
- [ ] **分析章节存在**：至少识别 3-5 个关键挑战点
- [ ] **解决方案章节存在**：针对主要挑战点提供解决方案
- [ ] **待确认项章节存在**：包含技术、业务、资源等多维度待确认项
- [ ] **Tasks/实施计划存在**：任务分解到可执行级别

**检查标准**：
```markdown
不合格：缺少任意一个章节 ❌
合格：所有章节都存在 ✅
优秀：各章节内容充实，逻辑清晰 ⭐
```

### 1.2 目录层级规范性

- [ ] **一级标题完整**：五个章节都有对应的一级标题
- [ ] **二级标题合理使用**：章节内使用二级标题组织内容
- [ ] **三级标题控制**：需求章节可使用三级标题，其他章节尽量避免

**检查要点**：
```markdown
✅ 合格示例：
## 3. 解决方案

### 3.1 缓存方案
### 3.2 消息队列方案

❌ 不合格示例：
## 3. 解决方案

### 3.1 缓存方案

#### 3.1.1 缓存选型
#### 3.1.2 缓存配置  ← 三级标题应避免
#### 3.1.3 缓存实现  ← 技术细节过多
```

### 1.3 配套文件结构

- [ ] **目录结构清晰**：
  ```
  SKILL.md                          # 主文件
  references/
    ├── document-structure.md       # 结构指南
    ├── style-guide.md              # 风格指南
    ├── quality-checklist.md        # 质量清单
    └── diagram-tools.md            # 图表工具指南
  assets/
    ├── *.drawio                    # 架构图源文件
    ├── *.png                       # 架构图预览
    └── *-c4-*.md                   # C4 模型文件
  ```

## 内容完整性检查

### 2.1 需求章节检查

#### 2.1.1 业务背景完整性
- [ ] **现状描述清晰**：描述当前系统的状态、存在的问题
- [ ] **业务目标明确**：有可衡量的业务指标
- [ ] **范围定义清晰**：明确项目的边界
- [ ] **利益相关方标识**：识别主要利益相关方

**检查问题**：
```markdown
□ 是否描述了业务痛点？
□ 是否有明确的业务指标（如：日单量提升 50%）？
□ 是否定义了项目范围（什么做、什么不做）？
□ 是否识别了关键利益相关方？
```

#### 2.1.2 功能需求完整性
- [ ] **功能点列举完整**：覆盖核心功能和辅助功能
- [ ] **功能描述清晰**：每个功能点都有简要说明
- [ ] **业务规则说明**：关键业务规则有说明
- [ ] **用例场景覆盖**：有典型用例场景描述

**检查标准**：
```markdown
不合格：只列出功能名称，无描述 ❌
合格：功能点 + 简要描述 ✅
优秀：功能点 + 详细描述 + 业务规则 + 用例 ⭐
```

#### 2.1.3 非功能需求完整性
- [ ] **性能指标明确**：有具体的性能指标（QPS、响应时间等）
- [ ] **可用性要求**：有可用性指标（99.9%、99.99%）
- [ ] **安全要求**：有安全相关的需求说明
- [ ] **其他要求**：可扩展性、可维护性等

**常见问题**：
```markdown
❌ 模糊表述："性能要好"、"系统要稳定"、"支持高并发"
✅ 明确表述："QPS 5000+"、"可用性 99.9%"、"响应时间 P99 < 200ms"
```

#### 2.1.4 约束条件完整性
- [ ] **时间约束**：有明确的时间要求
- [ ] **资源约束**：人力、预算等限制
- [ ] **技术约束**：技术栈、合规要求等
- [ ] **假设条件**：明确的前提假设

**缺失风险**：
```markdown
若缺少约束条件，可能导致：
- 方案设计与实际情况不符
- 资源估算不准确
- 技术选型不合理
- 计划制定不可行
```

### 2.2 分析章节检查

#### 2.2.1 挑战点识别充分性
- [ ] **挑战点数量充足**：至少识别 3-5 个关键挑战
- [ ] **覆盖不同维度**：技术、业务、资源等
- [ ] **具有针对性**：针对性强，与需求对应
- [ ] **分析深入**：有根因分析，不仅列症状

**评估标准**：
```markdown
不足：仅列出 1-2 个表面问题 ❌
合格：识别 3-5 个关键挑战 ✅
优秀：全面分析，有数据支撑，层次分明 ⭐
```

#### 2.2.2 挑战点描述质量
- [ ] **描述清晰**：问题现象清晰可理解
- [ ] **影响说明**：说明影响范围和程度
- [ ] **严重程度评估**：有严重等级（高/中/低）
- [ ] **数据支撑**：有现状数据支撑

**检查模板**：
```markdown
### X.X 挑战描述

**问题现象**：【清晰描述问题表现】

**影响范围**：【影响的业务、用户、系统】

**严重程度**：【高/中/低 + 原因】

**相关需求**：【追溯到需求章节】

**现状数据**：【数据支撑】
```

**示例**：
```markdown
❌ 不合格：

### 2.1 性能问题
数据库很慢，影响用户体验。

✅ 合格：

### 2.1 数据库性能瓶颈

**问题现象**：
- 高峰期数据库 CPU 使用率 95%+
- 订单查询平均响应时间 2-3 秒
- 订单创建超时率 5%

**影响范围**：
- 用户下单体验，转化率下降
- 大促期间可能导致系统崩溃

**严重程度**：高

**相关需求**：1.3.1 性能需求

**现状数据**：
- 当前 QPS：1000
- 业务需求 QPS：5000
- 单表数据量：5000 万条
```

#### 2.2.3 逻辑关系清晰性
- [ ] **与需求对应**：每个挑战都能追溯到需求
- [ ] **主次分明**：区分主要矛盾和次要矛盾
- [ ] **逻辑连贯**：按照重要性或逻辑顺序排列

### 2.3 解决方案章节检查

#### 2.3.1 方案完整性
- [ ] **覆盖关键挑战**：主要挑战点都有对应方案
- [ ] **未覆盖项处理**：未覆盖的挑战移入待确认项
- [ ] **方案分层**：有整体架构和具体实现方案

**检查方法**：
```markdown
建立对照表：

| 挑战点 | 严重度 | 是否有方案 | 章节位置 |
|--------|--------|------------|----------|
| 性能瓶颈 | 高 | ✓ | 3.1 |
| 系统耦合 | 中 | ✓ | 3.2 |
| 数据增长 | 高 | ✓ | 3.3 |
| 规则复杂 | 中 | ✗ | 4.1.1 |
```

#### 2.3.2 方案合理性
- [ ] **可行性**：方案在当前约束下可行
- [ ] **有效性**：能够有效解决对应挑战
- [ ] **先进性**：采用业界成熟方案
- [ ] **成本效益**：投入产出比合理

**评估维度**：
```markdown
□ 技术可行性：团队技术栈匹配度
□ 实施难度：是否需要在关键路径上攻克技术难点
□ 成本投入：硬件、软件、人力成本是否合理
□ 风险程度：方案本身的风险是否可控
□ 运维成本：后续维护的复杂度
```

#### 2.3.3 方案对比充分性（如适用）
- [ ] **多方案对比**：至少对比 2-3 个可选方案
- [ ] **维度全面**：功能性、性能、成本、风险等
- [ ] **客观公正**：优缺点分析客观
- [ ] **决策依据充分**：推荐方案有充分理由

**对比模板**：
```markdown
| 方案 | 描述 | 优点 | 缺点 | 成本 | 推荐度 |
|------|------|------|------|------|--------|
| A | ... | ... | ... | ... | ⭐⭐⭐ |
| B | ... | ... | ... | ... | ⭐⭐⭐⭐⭐ |
| C | ... | ... | ... | ... | ⭐⭐ |
```

#### 2.3.4 详略控制合理性
- [ ] **重点突出**：重点描述"是什么"和"为什么"
- [ ] **避免细节**：没有详细代码和配置
- [ ] **技术要点适度**：关键技术点有简要说明
- [ ] **引用配套文档**：细节在配套文档中

**检查方法**：
```markdown
□ 是否包含完整代码实现？→ 如有，移到详细设计文档
□ 是否包含详细配置参数？→ 如有，简化为关键参数
□ 是否有步骤化的操作指南？→ 如有，移到运维文档
□ 是否引用配套文档？→ 应该有引用说明
```

### 2.4 待确认项章节检查

#### 2.4.1 待确认项数量
- [ ] **数量适中**：通常 5-15 个待确认项
- [ ] **类型分散**：涵盖技术、业务、资源等维度
- [ ] **重要性均衡**：包含关键决策点

**评估标准**：
```markdown
过少：< 3 个 → 可能隐藏了待确认问题 ❌
适中：5-15 个 → 实事求是，考虑充分 ✅
过多：> 20 个 → 可能方案不成熟，需要确认太多 ⚠️
```

#### 2.4.2 待确认项质量
- [ ] **描述清晰**：问题描述清楚，不模糊
- [ ] **影响分析**：说明影响范围和程度
- [ ] **可选方案**：提供可选方案或分析思路
- [ ] **推进建议**：有推进确认的建议
- [ ] **时间计划**：有预期解决时间
- [ ] **责任明确**：有相关方或责任人

**质量模板**：
```markdown
### X.X 待确认事项

**待确认内容**：【清晰描述问题】

**影响分析**：【影响的范围】

**可选方案**：【如有】

**推进建议**：【建议的确认方式】

**相关方**：【决策方】

**目标时间**：【预期时间】
```

#### 2.4.3 与方案章节的一致性
- [ ] **无已解决项**：待确认项中没有已解决的问题
- [ ] **标注对应关系**：在解决方案中标注哪些挑战需要待确认
- [ ] **闭环跟踪**：每个待确认项都能追溯到具体的需求或挑战

### 2.5 Tasks/实施计划检查

#### 2.5.1 任务分解完整性
- [ ] **分解充分**：任务粒度适中（通常 1-3 人日）
- [ ] **覆盖全流程**：从需求到上线完整覆盖
- [ ] **依赖清晰**：任务间依赖关系明确
- [ ] **责任明确**：每个任务都有负责人

**检查标准**：
```markdown
不合格：任务粒度太大（如"完成开发"）❌
合格：任务分解清晰，可执行 ✅
优秀：任务分解细致，有完整的依赖图 ⭐
```

#### 2.5.2 时间估算合理性
- [ ] **估算依据**：有估算依据（历史数据、经验等）
- [ ] **Buffer 考虑**：考虑了风险 buffer（通常 20-30%）
- [ ] **里程碑合理**：关键里程碑设置合理
- [ ] **时间节点明确**：具体日期，非模糊时间

**验证方法**：
```markdown
□ 与历史项目对比：类似任务历史耗时多少？
□ 与团队确认：负责人认可时间估算？
□ 考虑风险：是否有 buffer 应对延期？
□ 关键路径分析：关键路径是否合理？
```

#### 2.5.3 风险评估充分性
- [ ] **风险识别全面**：识别技术、业务、管理风险
- [ ] **风险等级评估**：有正确的概率和影响评估
- [ ] **应对措施具体**：有具体的应对方案
- [ ] **监控指标明确**：有监控指标和触发条件

**风险矩阵模板**：
```markdown
| 风险描述 | 可能性 | 影响 | 等级 | 应对措施 | 监控指标 | 负责人 |
|----------|--------|------|------|----------|----------|--------|
| ... | ... | ... | ... | ... | ... | ... |
```

#### 2.5.4 资源规划合理性
- [ ] **人力资源充足**：人员配置满足任务需求
- [ ] **技能匹配**：团队技能与任务相匹配
- [ ] **环境资源**：测试、生产环境资源到位
- [ ] **预算合理**：预算估算合理且有依据

## 逻辑闭环检查

### 3.1 需求 → 分析闭环

- [ ] **每个分析都有需求来源**：检查是否能追溯到具体需求
- [ ] **需求全覆盖**：所有重要需求都有对应的分析
- [ ] **分析深度适中**：既不过度分析，也不遗漏关键点

**验证方法**：
```markdown
建立追踪矩阵：

| 需求 ID | 需求描述 | 相关分析 | 分析章节 | 比重 |
|---------|----------|----------|----------|------|
| R-001 | 支持 5000 TPS | 性能瓶颈分析 | 2.1 | 30% |
| R-002 | 99.9% 可用性 | 单点故障分析 | 2.2 | 25% |
... | ... | ... | ... | ... |
```

### 3.2 分析 → 方案闭环

- [ ] **每个挑战都有应对方案**：所有重要挑战都有解决方案
- [ ] **无对应方案项处理**：无方案的移入待确认项
- [ ] **方案针对性**：方案明确针对特定挑战

**验证方法**：
```markdown
建立对应关系：

| 挑战 ID | 挑战描述 | 严重度 | 解决方案 | 章节 | 有效性评估 |
|---------|----------|--------|----------|------|------------|
| C-001 | 数据库性能瓶颈 | 高 | 分库分表 | 3.1 | ⭐⭐⭐⭐⭐ |
| C-002 | 系统耦合严重 | 中 | 微服务拆分 | 3.2 | ⭐⭐⭐⭐ |
| C-003 | 规则复杂多变 | 中 | 待确认 | 4.1.1 | 待评估 |
```

### 3.3 方案 → 待确认项闭环

- [ ] **方案中未解决问题明确**：方案中注明哪些需要进一步确认
- [ ] **待确认项链接**：在方案中链接到待确认项章节
- [ ] **闭环确认**：方案评审时讨论待确认项

**示例**：
```markdown
### 3.3 规则引擎方案

🔍 **待讨论项**（详见章节 4.1.1）

考虑到规则的复杂性和多变性，建议引入规则引擎。
目前有两种可选方案：...

【此处提供对比，但明确标注需要进一步确认】
```

### 3.4 方案 → Tasks 闭环

- [ ] **已确认方案有对应任务**：每个确定方案都有实施任务
- [ ] **任务依赖合理**：任务之间的依赖关系正确
- [ ] **资源匹配**：任务与资源分配相匹配

**验证方法**：
```markdown
建立任务追踪：

| 方案 | 任务 ID | 任务描述 | 负责人 | 工作量 | 时间 |
|------|---------|----------|--------|--------|------|
| 3.1 | T-001 | 分库分表 | DBA | 10d | 2/1-2/10 |
| 3.2 | T-002 | 微服务拆分 | 开发 | 15d | 2/5-2/20 |
```

### 3.5 全文一致性检查

#### 3.5.1 数据一致性
- [ ] **性能指标一致**：全文性能指标不矛盾
- [ ] **资源数据一致**：服务器数量、配置等数据一致
- [ ] **时间数据一致**：项目周期、里程碑时间一致
- [ ] **人员数据一致**：团队规模、角色分工一致

**常见错误**：
```markdown
❌ 不一致示例：
- 需求章节：需要支持 QPS 10000
- 目标章节：设计目标 QPS 5000
- 方案章节：方案支持 QPS 20000
- 计划章节：压测目标 QPS 15000

✅ 正确做法：
- 需求：业务需求 QPS 10000
- 目标：设计目标 QPS 12000（留有余量）
- 方案：方案支持 QPS 12000
- 计划：压测目标 QPS 12000
```

#### 3.5.2 术语一致性
- [ ] **术语统一**：同一概念用词一致
- [ ] **缩写规范**：缩写有全称说明
- [ ] **避免歧义**：避免一词多义或多词一义

**检查清单**：
```markdown
□ Redis（不使用 redis、RedisServer、缓存服务器等混用）
□ 分库分表（不使用分片、Sharding 等混用）
□ 消息队列（不使用 MQ、消息中间件等混用）
□ 微服务（不使用微服务架构、微服务化等混用）
```

#### 3.5.3 逻辑一致性
- [ ] **因果关系**：因果关系正确
- [ ] **前后呼应**：前后文有呼应
- [ ] **论证充分**：论证过程充分，无逻辑跳跃

## 技术质量检查

### 4.1 技术选型合理性

#### 4.1.1 选型依据充分性
- [ ] **现状分析**：对现状有充分分析
- [ ] **选型对比**：对比了多种技术方案
- [ ] **权衡考虑**：考虑了功能、性能、成本、风险
- [ ] **决策理由**：有明确的决策理由

**选型矩阵**：
```markdown
| 技术方案 | 功能匹配 | 性能 | 成本 | 风险 | 成熟度 | 总分 |
|----------|----------|------|------|------|--------|------|
| 方案 A | 9 | 8 | 9 | 7 | 9 | 42 |
| 方案 B | 8 | 9 | 7 | 8 | 8 | 40 |
| 方案 C | 7 | 7 | 8 | 9 | 7 | 38 |
```

#### 4.1.2 技术先进性与成熟性平衡
- [ ] **不过于保守**：不是全部使用最保守方案
- [ ] **不过于激进**：避免使用未经验证的新技术
- [ ] **合理平衡**：主流技术为主，适当引入新技术

**性能指标模板**：
```markdown
QPS（Queries Per Second）：
- 业务目标：5000 QPS
- 峰值估计：10000 QPS（考虑 2 倍峰值系数）
- 设计目标：12000 QPS（预留 20% 余量）

响应时间（Response Time）：
- P50 目标：50ms（50% 请求）
- P90 目标：100ms（90% 请求）
- P99 目标：200ms（99% 请求）
- P999 目标：500ms（99.9% 请求）
```

#### 5.2.2 可用性指标
- [ ] **目标明确**：明确的可用性目标
- [ ] **计算方式**：有可用性计算公式
- [ ] **符合预期**：符合业务预期

**可用性计算**：
```markdown
可用性 = (总时间 - 故障时间) / 总时间 × 100%

目标：99.9%（3个9）
含义：每月故障时间 < 43.8 分钟

测量：
- MTBF（Mean Time Between Failures）：平均故障间隔时间
- MTTR（Mean Time To Repair）：平均修复时间

公式：
可用性 = MTBF / (MTBF + MTTR)
```

#### 5.2.3 容量数据

- [ ] **扩展空间**：考虑了未来增长

**容量估算参考**：
```markdown
存储容量 = 单条数据大小 × 数据量 × (1 + 冗余系数)

示例：
- 订单数据：
  - 单条大小：2KB
  - 日增量：10 万
  - 保留期：1 年
  - 计算：2KB × 10万 × 365 × 1.5（冗余）≈ 1TB
```

## 常见问题与解决

### 问题 1：待确认项太多（> 20 个）

**原因**：
- 方案不成熟，需要确认的问题太多
- 过于谨慎，把太多细节都列入待确认

**解决方案**：
1. 区分"待确认"和"待细化"
2. 待确认项聚焦关键决策点
3. 细节问题在施工阶段确认

### 问题 2：方案过度详细

**原因**：
- 混淆概要设计和详细设计
- 担心被认为不够详细

**解决方案**：
1. 牢记"概要"定位
2. 详细内容移到配套文档
3. 提供配套文档清单

### 问题 3：逻辑闭环不严

**原因**：
- 写作时未建立追踪关系
- 修改时未同步更新

**解决方案**：
1. 使用对照表追踪（需求 → 分析 → 方案）
2. 每次修改后检查一致性
3. 使用本检查清单逐项核对

### 问题 4：数据支撑不足

**原因**：
- 缺少监控数据
- 时间紧迫未收集数据

**解决方案**：
1. 建设基础监控
2. 进行 POC 测试
3. 引用业界公开数据
4. 标注数据待收集（Todo）
