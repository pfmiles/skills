## 整体风格定位

### 核心特征

| 特征维度 | 要求 | 说明 | 示例 |
|----------|------|------|------|
| **朴素性** | 平实自然 | 避免华丽辞藻和修饰 | "系统通过 Redis 缓存提升查询性能" |
| **准确性** | 事实准确 | 数据、指标、描述准确 | "QPS 从 1000 提升至 5000" |
| **条理性** | 逻辑清晰 | 层次分明，因果关系明确 | "由于 A，导致 B，因此需要 C" |
| **简洁性** | 简洁精炼 | 无冗余描述，每句话传达有效信息 | 删除"很显然"、"大家都知道"等冗余词 |
| **专业性** | 技术专业 | 使用准确的技术术语和概念 | "使用 TCC 模式解决分布式事务" |

#### ✅ 符合风格的写法
```markdown

1. 峰值 TPS 仅 1000，无法支撑大促期间 5000TPS 的目标
2. 数据库单表数据量超过 5000 万，查询性能持续下降

## 详略控制原则

### 核心原则："三问"原则

在撰写每一部分内容时，问自己三个问题：

1. **WHAT**：这部分要传达的核心信息是什么？
2. **WHY**：读者为什么需要知道这些信息？
3. **HOW MUCH**：需要多详细的程度？

### 详略分区控制

| 章节 | 详略程度 | 原因 | 具体做法 |
|------|----------|------|----------|
| **需求** | ⭐⭐⭐⭐⭐ | 基础信息，需要充分理解 | 充分描述业务背景、功能、非功能需求 |
| **分析** | ⭐⭐⭐⭐ | 提炼要点，支撑方案 | 提炼关键挑战，不过度展开 |
| **解决方案** | ⭐⭐⭐ | 概要为王，避免细节 | 描述设计思路，不编写代码 |
| **待确认项** | ⭐⭐⭐⭐ | 清晰明确，避免模糊 | 每个项都有明确的内容和影响 |
| **实施计划** | ⭐⭐⭐⭐⭐ | 具体可执行 | 任务分解细致，时间明确 |

### "三描述"原则

#### 1. 重点描述："是什么"（What）
**核心**：方案的整体设计和架构思路

**示例**：
```markdown
### 3.1 缓存方案

**是什么**：
采用 Redis Cluster 作为分布式缓存，通过读写分离和热点数据缓存，提升系统读取性能 5-10 倍。

**架构**：
- 应用层集成 Redis 客户端
- Redis 3 主 3 从集群部署
- 热点数据 TTL 设置 10 分钟
- 使用 Redisson 客户端连接池
```

#### 2. 重点描述："为什么"（Why）
**核心**：设计决策的理由和依据

**示例**：
```markdown
#### 为什么采用 Redis：

1. **性能要求**：需要支撑 10万+ QPS，MySQL 无法满足

2. **数据特征**：
   - 用户配置、商品信息等读多写少
   - 对实时性要求不高（可接受 10 分钟延迟）
   - 数据量 20GB 以内

3. **技术成熟度**：
   - Redis 在业界应用广泛，有大量成功案例
   - 团队有相关使用经验
   - 社区活跃，文档完善

4. **成本效益**：
   - 相比增加 MySQL 从库，成本更低
   - 性能提升显著，ROI 高
```

#### 3. 适度描述："怎么做"（How）
**核心**：关键技术点的实现思路

**示例**：
```markdown
#### 关键技术实现思路：

**1. 缓存更新策略**
- 写操作：更新数据库后，同步更新缓存（Cache Aside 模式）
- 读操作：先读缓存，缓存未命中再读数据库并写入缓存

**2. 缓存穿透保护**
- 布隆过滤器拦截不存在的 key
- 空值缓存（short duration）

**3. 热点数据识别**
- 基于访问频率统计前 1000 个热点 key
- 自动缓存热点数据
```

#### 避免描述：过度技术细节
**不推荐**：详细代码、完整配置、分步操作指南

**错误示例**：
```markdown

### 使用示例：

```java
@Service
public class OrderCacheService {

    @Autowired
    private RedissonClient redissonClient;

    @Autowired
    private OrderMapper orderMapper;

    private static final String ORDER_CACHE_KEY = "order:cache:";
    private static final long CACHE_TTL = 10; // 分钟

    /**
     * 根据订单 ID 查询订单信息（带缓存）
     */
    public Order getOrderById(Long orderId) {
        String cacheKey = ORDER_CACHE_KEY + orderId;

        // 先从缓存获取
        RBucket<Order> bucket = redissonClient.getBucket(cacheKey);
        Order order = bucket.get();

        if (order != null) {
            log.info("从缓存获取订单: orderId={}", orderId);
            return order;
        }

        // 缓存未命中，查询数据库
        order = orderMapper.selectById(orderId);

        if (order != null) {
            // 写入缓存
            bucket.set(order, CACHE_TTL, TimeUnit.MINUTES);
            log.info("缓存未命中，查询数据库并写入缓存: orderId={}", orderId);
        } else {
            // 防止缓存穿透，写入空值
            bucket.set(new Order(), 1, TimeUnit.MINUTES);
            log.warn("订单不存在，写入空缓存: orderId={}", orderId);
        }

        return order;
    }

    /**
     * 更新订单信息（更新缓存）
     */
    public void updateOrder(Order order) {
        // 更新数据库
        orderMapper.updateById(order);

        // 更新缓存
        String cacheKey = ORDER_CACHE_KEY + order.getId();
        RBucket<Order> bucket = redissonClient.getBucket(cacheKey);
        bucket.set(order, CACHE_TTL, TimeUnit.MINUTES);

        log.info("更新订单并刷新缓存: orderId={}", order.getId());
    }

    /**
     * 删除订单（删除缓存）
     */
    public void deleteOrder(Long orderId) {
        // 删除数据库记录
        orderMapper.deleteById(orderId);

        // 删除缓存
        String cacheKey = ORDER_CACHE_KEY + orderId;
        redissonClient.getBucket(cacheKey).delete();

        log.info("删除订单及缓存: orderId={}", orderId);
    }
}
```

### 详略判断标准

## 段落与结构

### 段落长度控制

#### 推荐长度：3-8 行
**原因**：
- 太短：信息碎片化，缺乏上下文
- 太长：阅读疲劳，重点不突出
- 适中：重点清晰，易于理解

**示例**：
```markdown
✅ 段落长度适中：

### 3.1 缓存方案

采用 Redis Cluster 作为分布式缓存（What）。需要支撑 10万+ QPS 的查询需求，而 MySQL 无法满足（Why）。订单、用户等配置信息读多写少，且对实时性要求不高，适合缓存场景（Why）。相比数据库扩容，成本更低、效果更显著（Why）。

```java
// 查询订单（带缓存）
public Order getOrder(Long orderId) {
    // 1. 先查缓存
    Order order = redis.get("order:" + orderId);
    if (order != null) {
        return order; // 缓存命中
    }

    // 2. 缓存未命中，查数据库
    order = db.query("SELECT * FROM order WHERE id = ?", orderId);

    // 3. 写入缓存（10 分钟过期）
    if (order != null) {
        redis.setex("order:" + orderId, 600, order);
    } else {
        // 4. 空值缓存（防止穿透）
        redis.setex("order:" + orderId, 60, "EMPTY");
    }

    return order;
}
```

- 梳理目标值：业务目标 10000，设计方案留有余地 20000，
  压测验证 15000（确保满足业务目标）
- 明确各数值的含义和关系

#### 逻辑闭环错误
```markdown
❌ 错误：待确认项中有已解决的问题

待确认项：
- 4.1.1 数据库选型（已经在 3.1 给出明确方案）
- 4.1.2 缓存策略（已经在 3.2 明确描述）
```

**修正方法**：
- 已解决的从待确认项移除
- 仍不确定的留在待确认项
- 说明已经做了什么分析，还需要确认什么

#### 详略颠倒错误
```markdown
❌ 错误：需求章节过于简单，解决方案过于详细

需求章节：
- 需要提升性能
- 需要支持更多用户

解决方案章节：
- Redis 配置详解（200 行）
- 完整代码实现（500 行）
- 详细部署步骤（分 10 步详细描述）
```

**修正方法**：
- 需求章节充分展开（业务场景、痛点、目标）
- 解决方案章节只保留设计思路（删除细节）
- 细节移到配套文档或 TODO 后续
